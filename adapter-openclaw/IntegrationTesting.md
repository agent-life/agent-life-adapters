# Integration Testing Strategy

## Scope

This document defines the integration testing approach for the agent-life-adapters
workspace. It covers adapter-level, CLI-level, and service-level testing — the
latter designed to be activated when the agent-life sync service goes live.

Tests in this document are **integration tests** — they exercise multiple modules
together, use the filesystem, and validate end-to-end data paths. They complement
the existing 100 unit tests (alf-core 78, alf-cli 22, adapter-openclaw 45) which
test individual functions in isolation.

All integration tests live in the `tests/` directory of the relevant crate.

---

## 1. Fixture Strategy

### 1.1 Canonical Fixtures

Create `tests/fixtures/` directories containing representative workspace snapshots
and pre-built `.alf` archives. These are checked into version control so tests are
deterministic and reproducible across machines.

```
adapter-openclaw/tests/fixtures/
├── minimal/                    # Bare minimum: SOUL.md only
│   └── SOUL.md
├── standard/                   # Typical workspace
│   ├── SOUL.md
│   ├── IDENTITY.md
│   ├── AGENTS.md
│   ├── USER.md
│   ├── MEMORY.md
│   ├── TOOLS.md
│   ├── HEARTBEAT.md
│   └── memory/
│       ├── 2026-01-15.md       # 3 H2 sections
│       ├── 2026-01-16.md       # 1 H2 section
│       ├── active-context.md
│       ├── project-clawsmith.md
│       └── gating-policies.md
├── community-patterns/         # Community memory extensions
│   ├── SOUL.md
│   └── memory/
│       ├── 2026-02-01.md       # Importance tags: [decision|i=0.9]
│       ├── active-context.md
│       └── facts.db            # SQLite (future; ignored for now)
├── empty/                      # Workspace dir exists but no files
├── cross-runtime.alf           # ALF archive with no raw/openclaw/ entries
└── golden/
    └── standard.alf            # Known-good export of standard/ fixture
```

### 1.2 Fixture Conventions

- **Deterministic content.** Fixture files contain hardcoded text. No timestamps
  derived from `now()` — use explicit dates like `2026-01-15`.
- **Golden files.** `golden/*.alf` archives are re-generated by a dedicated
  `update-golden` binary/script when the format changes. Tests compare against
  these to detect unintentional regressions.
- **Tempdir for outputs.** All test outputs write to `tempfile::TempDir`. Tests
  never write to the source tree.

## 2. Adapter Integration Tests

Crate: `adapter-openclaw/tests/`

These tests exercise the full export → archive → import pipeline through the
public `OpenClawAdapter` API.

### 2.1 Round-Trip Fidelity (`round_trip.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `round_trip_standard_workspace` | Export `standard/` → import to fresh dir | All `.md` files byte-identical via `diff -r` |
| `round_trip_minimal_workspace` | Export `minimal/` → import | SOUL.md restored, no extra files created |
| `round_trip_community_patterns` | Export `community-patterns/` → import | Importance tags, active-context preserved |
| `round_trip_preserves_file_permissions` | Export → import | File permissions match (mode, not ownership) |
| `round_trip_unicode_content` | Workspace with CJK, emoji, RTL text | Content byte-identical after round-trip |
| `round_trip_large_daily_log` | Daily log with 200+ H2 sections | All sections present, order preserved |

Implementation pattern:
```rust
#[test]
fn round_trip_standard_workspace() {
    let fixture = Path::new("tests/fixtures/standard");
    let tmp = TempDir::new().unwrap();
    let alf_path = tmp.path().join("export.alf");
    let restored = tmp.path().join("restored");

    let adapter = OpenClawAdapter;
    let export_report = adapter.export(fixture, &alf_path).unwrap();
    let import_report = adapter.import(&alf_path, &restored).unwrap();

    assert_eq!(export_report.memory_records, import_report.memory_records);
    assert_files_match(fixture, &restored, &ROUND_TRIP_FILES);
}
```

### 2.2 Export Correctness (`export_correctness.rs`)

Test that exports produce valid, well-structured ALF archives by opening the
result with `AlfReader` and inspecting structured fields.

| Test | Description | Assertion |
|------|-------------|-----------|
| `manifest_metadata` | Export standard workspace | Manifest has `source_runtime = "openclaw"`, correct agent name, valid UUID |
| `memory_record_ids_deterministic` | Export same workspace twice | Every record UUID identical between runs |
| `memory_type_classification` | Export standard workspace | Daily logs → Episodic, MEMORY.md → Semantic, active-context → Summary, gating-policies → Procedural |
| `namespace_classification` | Export standard workspace | Correct namespace per file type |
| `daily_log_observed_at` | Export workspace with dated logs | `observed_at` matches filename date |
| `importance_tags_extracted` | Export community-patterns | `confidence`, `category`, `tags` populated from `[tag\|i=N.N]` |
| `identity_layer_populated` | Export workspace with SOUL + IDENTITY + AGENTS | All three prose fields present |
| `principals_layer_populated` | Export workspace with USER.md | One Human principal with name + timezone |
| `raw_sources_included` | Export standard workspace | Every `.md` file present under `raw/openclaw/` |
| `partition_assignment` | Export workspace spanning multiple quarters | Records grouped into correct quarterly partitions |
| `empty_workspace_valid` | Export empty workspace | Valid archive with 0 memory records, no identity/principals |
| `agent_id_persisted` | Export writes `.alf-agent-id` | File exists, content is valid UUID, same UUID in manifest |

### 2.3 Import Correctness (`import_correctness.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `import_creates_directory_structure` | Import to nonexistent path | `memory/` subdir created |
| `import_writes_agent_id` | Import any archive | `.alf-agent-id` written with matching UUID |
| `import_credentials_warning` | Import archive with credentials | Warning mentions re-authentication |
| `import_overwrites_existing` | Import into a workspace that already has files | Files replaced (not merged) |

### 2.4 Cross-Runtime Migration (`cross_import.rs`)

Test importing an ALF archive that was **not** created by the OpenClaw adapter
(no `raw/openclaw/` entries). The adapter must reconstruct workspace files from
structured data.

| Test | Description | Assertion |
|------|-------------|-----------|
| `cross_import_identity_to_soul` | Import ALF with identity prose but no raw sources | `SOUL.md` created from `prose.soul` |
| `cross_import_principals_to_user` | Import ALF with principals prose | `USER.md` created from `prose.user_profile` |
| `cross_import_memory_daily` | Import ALF with daily namespace records | `memory/YYYY-MM-DD.md` files created, grouped by `observed_at` |
| `cross_import_memory_curated` | Import ALF with curated namespace records | `MEMORY.md` created from curated records |
| `cross_import_structured_fallback` | Import ALF with structured identity but no prose | Minimal `SOUL.md` synthesized from name + role |
| `cross_import_warning_emitted` | Import any cross-runtime archive | Warning about reconstruction present in report |

### 2.5 Delta Stability (`delta_stability.rs`)

Verify that the deterministic ID generation enables correct delta computation
when the same workspace is exported at different points in time.

| Test | Description | Assertion |
|------|-------------|-----------|
| `no_changes_empty_delta` | Export workspace twice (identical) | `compute_delta(old, new)` returns empty |
| `new_section_detected` | Add an H2 section to a daily log, re-export | Delta contains one `Added` entry |
| `modified_section_detected` | Edit text within an H2 section, re-export | Delta contains one `Modified` entry with correct ID |
| `deleted_section_detected` | Remove an H2 section, re-export | Delta contains one `Removed` entry |
| `renamed_file_produces_new_ids` | Rename a daily log file, re-export | Old IDs appear as Removed, new IDs as Added |

## 3. Validation Integration Tests

Crate: `alf-core/tests/` or `adapter-openclaw/tests/`

Run the full validation pipeline against adapter-produced archives.

| Test | Description | Assertion |
|------|-------------|-----------|
| `exported_archive_passes_validation` | Export standard workspace, run `validate_archive()` | `report.is_valid() == true`, no errors |
| `exported_archive_no_warnings` | Same as above | No warnings either (adapter output should be clean) |
| `corrupted_archive_detected` | Tamper with a byte in the ZIP, validate | Error reported |
| `missing_manifest_detected` | Remove manifest.json from ZIP, validate | Error reported |
| `record_count_mismatch_detected` | Edit manifest to wrong record_count, validate | Error reported |

## 4. CLI End-to-End Tests

Crate: `alf-cli/tests/`

These tests invoke the `alf` binary as a subprocess and verify stdout, stderr,
and exit codes. They test the full user-facing command flow.

### 4.1 Infrastructure

```rust
fn alf_cmd() -> Command {
    Command::cargo_bin("alf").expect("binary not found")
}

fn assert_success(cmd: &mut Command) -> String {
    let output = cmd.output().expect("failed to execute");
    assert!(output.status.success(), "stderr: {}", String::from_utf8_lossy(&output.stderr));
    String::from_utf8_lossy(&output.stdout).to_string()
}
```

Add `assert_cmd` and `predicates` to `[dev-dependencies]` for ergonomic
assertions.

### 4.2 Export Command (`cli_export.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `export_success` | `alf export --runtime openclaw --workspace <fixture> --output <tmp>` | Exit 0, `.alf` file created, summary printed |
| `export_unknown_runtime` | `alf export --runtime unknown ...` | Exit non-zero, stderr contains "unsupported runtime" |
| `export_missing_workspace` | `alf export --runtime openclaw --workspace /nonexistent ...` | Exit non-zero, stderr contains error |
| `export_default_output_name` | `alf export --runtime openclaw --workspace <fixture>` (no --output) | File created with default naming convention |

### 4.3 Import Command (`cli_import.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `import_success` | Export then `alf import --runtime openclaw --alf-file <file> --workspace <tmp>` | Exit 0, workspace created |
| `import_missing_file` | `alf import --runtime openclaw --alf-file /nonexistent ...` | Exit non-zero |
| `import_to_existing_workspace` | Import twice to same dir | Second import succeeds (overwrite) |

### 4.4 Validate Command (`cli_validate.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `validate_valid_archive` | Export then `alf validate <file>` | Exit 0, "valid" in stdout |
| `validate_corrupt_archive` | Feed a non-ZIP file | Exit non-zero, error in stderr |

### 4.5 Full Pipeline (`cli_pipeline.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `export_validate_import` | `alf export` → `alf validate` → `alf import` | All three succeed, workspace files match |
| `export_diff_export` | Export, modify workspace, export again | Second archive differs, both valid |

## 5. Service Integration Tests (Activate When Live)

These tests are designed to run against the agent-life sync service. They are
gated behind a feature flag or environment variable so they don't run in offline
CI.

### 5.1 Gating Mechanism

```rust
fn skip_unless_service() {
    if std::env::var("ALF_SERVICE_URL").is_err() {
        eprintln!("Skipping: ALF_SERVICE_URL not set");
        return;
    }
}
```

Or use a Cargo feature:
```toml
[features]
service-tests = []

# Run with: cargo test --features service-tests
```

### 5.2 Auth Flow (`service_auth.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `login_valid_credentials` | `ApiClient::login()` with test credentials | Returns valid token |
| `login_invalid_credentials` | Login with bad password | Returns auth error, not panic |
| `token_refresh` | Login, wait, make authenticated call | Token auto-refreshed or error is clear |
| `token_expiry_handled` | Use an expired token | 401 returned, client handles gracefully |

### 5.3 Agent Registration (`service_registration.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `register_new_agent` | `register_agent()` with fresh UUID | Returns AgentInfo with matching ID |
| `register_duplicate_agent` | Register same UUID twice | Idempotent (returns existing) or clear error |
| `get_agent` | Register then `get_agent()` | Returns matching metadata |
| `get_nonexistent_agent` | `get_agent()` with random UUID | Returns 404/None, not crash |

### 5.4 Snapshot Upload/Download (`service_snapshot.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `upload_then_download` | Export workspace → upload → download → validate | Downloaded bytes match uploaded |
| `upload_large_archive` | Upload a 50MB+ archive | Succeeds within timeout |
| `download_validates` | Download snapshot, run `validate_archive()` | Valid archive |
| `upload_idempotent` | Upload same snapshot twice | No error, latest wins |

### 5.5 Delta Sync (`service_delta.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `upload_delta` | Export, modify, compute delta, upload | Returns DeltaInfo with sequence number |
| `list_deltas_since` | Upload 3 deltas, list since seq 1 | Returns 2 deltas |
| `download_and_apply_delta` | Upload delta, download, apply to local | Local state matches expected |
| `delta_ordering` | Upload deltas out of order | Service rejects or reorders correctly |
| `concurrent_delta_upload` | Two clients upload deltas simultaneously | Conflict detected or last-write-wins |

### 5.6 Sync Command E2E (`service_sync_e2e.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `first_sync_uploads_snapshot` | `alf sync` on fresh agent | Snapshot uploaded, local state updated |
| `subsequent_sync_uploads_delta` | Modify workspace, `alf sync` again | Delta uploaded (not full snapshot) |
| `sync_after_restore` | `alf restore`, modify, `alf sync` | Delta computed against restored baseline |
| `sync_no_changes` | `alf sync` with no workspace changes | No upload, "already up to date" message |

### 5.7 Restore Command E2E (`service_restore.rs`)

| Test | Description | Assertion |
|------|-------------|-----------|
| `restore_latest` | Upload snapshot, `alf restore` to fresh dir | Workspace matches original |
| `restore_with_deltas` | Snapshot + 3 deltas, restore | Final state includes all delta changes |
| `restore_to_existing_dir` | Restore into a non-empty workspace | Existing files replaced |
| `restore_no_snapshot` | Restore agent with no uploads | Clear error message |

## 6. Property-Based Tests

For additional confidence, consider property-based testing (using `proptest` or
`quickcheck`) for core invariants.

| Property | Module | Description |
|----------|--------|-------------|
| `split_join_identity` | `memory_parser` | For any Markdown string, splitting into sections and joining content reproduces the original (modulo trailing newlines) |
| `record_id_deterministic` | `memory_parser` | For any `(path, index)` pair, `record_id` always returns the same UUID |
| `export_import_record_count` | `export`/`import` | For any valid workspace, export record count equals import record count |
| `delta_apply_inverse` | `delta` | `apply_delta(old, compute_delta(old, new))` produces a set equivalent to `new` |
| `validation_never_panics` | `validation` | Feeding arbitrary bytes to validation functions produces errors, never panics |

## 8. CI Configuration

### 8.1 Offline Tests (Every PR)

```yaml
test-offline:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: dtolnay/rust-toolchain@stable
    - run: cargo test --workspace
    # Includes all unit tests + adapter/CLI integration tests
    # Excludes service tests (no ALF_SERVICE_URL set)
```

### 8.2 Service Tests (Nightly or Pre-Release)

```yaml
test-service:
  runs-on: ubuntu-latest
  environment: staging
  steps:
    - uses: actions/checkout@v4
    - uses: dtolnay/rust-toolchain@stable
    - run: cargo test --workspace --features service-tests
      env:
        ALF_SERVICE_URL: ${{ secrets.ALF_STAGING_URL }}
        ALF_TEST_TOKEN: ${{ secrets.ALF_STAGING_TOKEN }}
```

### 8.3 Golden File Updates

When the archive format changes intentionally:

```bash
cargo run --bin update-golden -- tests/fixtures/standard tests/fixtures/golden/standard.alf
```

Golden file changes must be committed alongside the format change in the same PR.

## 10. Implementation Priority

Start with the tests that catch the highest-impact bugs first:

1. **Round-trip fidelity** (§2.1) — if data is lost, everything else is moot
2. **Export correctness** (§2.2) — verify the mapping logic works correctly
3. **Cross-runtime migration** (§2.4) — the hardest code path, most likely to have bugs
4. **Delta stability** (§2.5) — critical for sync; ID determinism is a load-bearing invariant
5. **CLI end-to-end** (§4) — catches wiring issues between commands and adapter
6. **Validation integration** (§3) — safety net for malformed archives
7. **Service integration** (§5) — activate when API is available
8. **Property-based** (§6) — add incrementally for core invariants
